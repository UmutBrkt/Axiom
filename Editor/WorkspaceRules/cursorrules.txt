# Axiom — Agent Workspace Instructions

## Core Rules

1. **Report First, Act Second.** Before making any changes, run the appropriate Diagnostic Bridge tool to understand current state. Never modify blindly.
2. **Wiring via Editor Scripts.** Default to writing temporary [MenuItem] editor utility scripts for assigning SerializedProperty references. Never use MCP inspector interactions for wiring.
3. **Logging Over Browsing.** Prefer reading reports from AgentReports/ over browsing hierarchy or folders via MCP tools.
4. **Constrain Scans.** Always provide a root_path, object_names, or filter when using diagnostic tools. Never scan the entire scene or project unless explicitly asked.
5. **Validate on Reload.** After script changes, use `Axiom_Verify` with `{"operation": "compilation"}` — it automatically calls `AssetDatabase.Refresh(ForceUpdate)` and waits for compilation to finish before reporting. Confirm `Status: CLEAN` before proceeding.
6. **Use SerializedProperties.** Always use SerializedObject for edits to preserve Undo and Prefab overrides.
7. **Use AssetDatabase for Moves.** Never use filesystem operations to move assets.
8. **Prefer Build Profiles.** In Unity 6, use Build Profiles over global PlayerSettings when available.
9. **Modal Awareness.** Track whether editor is in Edit, Play, or Prefab Mode. Play Mode changes do not persist.
10. **State Persistence.** After completing work, update AgentReports/StatusUpdate.md with a summary.
11. **Respect Component Dependencies.** Before removing a component, verify no `RequireComponent` attribute on sibling components depends on it. `ComponentActions.RemoveComponent` does this automatically.
12. **Cache Reflection.** All `MethodInfo`/`FieldInfo` lookups for internal APIs must be cached as static fields, not re-reflected per call. See `LogMirror.cs` for the pattern.
13. **Snapshot Before Major Changes.** Before multi-step operations, save a SceneDiff snapshot for rollback: `SceneDiff.Execute(SceneDiffOperation.Snapshot, label: "before_feature_x")`.
14. **Temp Scripts Location.** All temporary editor scripts (pre-flight checks, verification, one-off utilities) must be created inside `Assets/Axiom/Editor/AgentBridge/Temp/`. Never place scripts at the project root or directly under `Assets/` — Unity cannot compile them there without an assembly definition. Delete temp scripts after use.
15. **Wait for Compilation.** After finishing a set of code changes (new scripts, modifications, deletions), wait for Unity to compile before attempting to execute any script. Never run a temp script immediately after creating it — Unity needs time to detect and compile the change. Do not assume compilation is instant.
16. **Verify Compilation via Axiom_Verify.** After any code change, confirm clean compilation by calling `Axiom_Verify` with `{"operation": "compilation"}`. This tool automatically triggers `AssetDatabase.Refresh(ForceUpdate)` so Unity detects file changes written by external agents, then waits up to 30s for compilation to finish before generating the report. **Without this refresh step, compilation checks report the PREVIOUS state — showing false CLEAN on uncompiled code.** Read the report and proceed only if `Status: CLEAN`. If `Status: HAS_ERRORS`, read error details and fix. Never trust IDE linter squiggles — only Unity's compiler output is authoritative.

17. **NEVER Use Shell Commands for Unity Exploration.** Do not use PowerShell, bash, cmd, or any terminal commands for Unity project discovery, asset searches, scene inspection, or hierarchy browsing. This includes but is not limited to:
    - `Get-ChildItem` / `dir` / `ls` to find assets
    - `Select-String` / `grep` / `findstr` to search scene YAML files
    - `Test-Path` to check asset existence
    - `Get-Content` to read `.unity` / `.prefab` / `.asset` files
    
    All discovery MUST go through Axiom diagnostic tools or `Unity_RunCommand` with `AssetDatabase` / `EditorSceneManager` APIs. Unity scene YAML stores references as GUIDs and fileIDs — shell text searches cannot resolve these to type names, component names, or asset references. Axiom tools resolve them internally, which is why they exist. **There are zero exceptions to this rule.**

18. **When Axiom Returns Unexpected Results, Stay in Framework.** If a diagnostic tool returns data you don't expect (wrong scene, empty results, scope mismatch), diagnose the problem using OTHER Axiom tools — do not fall back to shell commands or raw file access. Follow this recovery sequence:
    - **Wrong scene data?** → The target scene likely isn't loaded. Use `scene_actions` `manage_scene` `load` with the scene path, then re-run the diagnostic.
    - **Empty asset results?** → Check your filter parameters. `component_filter` matches Components on GameObjects, not asset types. For asset-type searches, use `asset_extension` (e.g. `".physicMaterial"`) or `assetType` (e.g. `"PhysicMaterial"`). If unsure, broaden the search or use `Unity_RunCommand` with `AssetDatabase.FindAssets("t:TypeName")`.
    - **Tool returns an error?** → Read the error message. The gateway returns descriptive errors including valid mode names. Do not guess modes by trial-and-error.
    - **Still stuck after two Axiom attempts?** → Read the tool's source file in `Assets/Axiom/Editor/AgentBridge/` to confirm valid parameters, modes, and expected behavior. Then retry with corrected parameters.
    
    The cost of one wasted Axiom call is far less than the cost of a shell-command spiral that produces no usable data. Always exhaust Axiom options before considering any alternative.

19. **Read Tool Schema Before Unfamiliar Calls.** Before calling any Axiom tool with parameters you haven't used before in this session, confirm the valid modes and parameter schema by ONE of:
    - Checking this `.cursorrules` file (the tool reference sections below list all modes and parameters)
    - Using `Axiom_Rules` (if available via MCP) to retrieve the operating rules
    - Reading the tool's source file (e.g. `Assets/Axiom/Editor/AgentBridge/Actions/SceneActions.cs`)
    
    Do NOT guess mode names or parameter structures. Three failed gateway calls with guessed parameters cost more tokens than one file read. The gateway validates strictly — `additionalProperties: false` means invented parameters are rejected, not silently ignored.

20. **Use the Right Execution Path.** Match the task to the correct execution mechanism:
    - **One-shot logic** (find assets, remove missing scripts, set a property, run a query): Use `Unity_RunCommand` with `IRunCommand.Execute()`. This compiles and runs inline — no menu click needed.
    - **Axiom diagnostic/action tools**: Use the `AgentBridgeGateway.Execute(json)` via `execute_script`. One call, no temp file.
    - **Persistent utilities** (setup scripts that wire multiple references, debug inspectors): Write a `[MenuItem]` script in `Assets/Axiom/Editor/AgentBridge/Temp/`. These require a menu click or `EditorApplication.ExecuteMenuItem()` to invoke.
    
    Do NOT write a `[MenuItem]` temp script for one-shot operations — `Unity_RunCommand` exists for exactly this purpose. Do NOT create a temp script, realize you can't invoke it, delete it, and then use `Unity_RunCommand`. Choose the right path first.

21. **Scene Loading Awareness.** Axiom diagnostic tools operate on scenes that are currently loaded in the Unity Editor. They cannot inspect scenes that exist only as `.unity` files on disk.
    - Before running diagnostics on a specific scene, confirm it is loaded. If `hierarchy_lens` returns data from a different scene than expected, the target scene is not loaded.
    - To load a scene: `SceneActions.ManageScene("load", scenePath: "Assets/Scenes/MyScene.unity", additive: true)`
    - To load via gateway: `{"tool": "scene_actions", "mode": "manage_scene", "scope": {"root_path": "Assets/Scenes/MyScene.unity", "object_names": ["load"]}}`
    - After loading, re-run the diagnostic tool. Do NOT attempt to read the `.unity` file directly via shell commands — YAML GUIDs are not human-interpretable and Axiom resolves them for you.

22. **No Silent Deviations.** If you are about to take an action that is not covered by the Axiom toolchain or the patterns in this file, STOP and inform the user before proceeding. Describe what you want to do, why the standard tools don't cover it, and ask for approval. Do not self-approve deviations. Do not rationalize shell commands as "faster" or "simpler" — they produce data that lacks Unity's internal resolution (GUIDs, SerializedObject state, asset pipeline metadata) and are therefore less accurate, not simpler.

## AgentBridge Gateway (PRIMARY INTERFACE)

The gateway is the preferred way to call Axiom tools. It accepts JSON commands and dispatches to the correct tool without requiring temp script compilation.

**Implementation:** `Assets/Axiom/Editor/AgentBridge/Core/AgentBridgeGateway.cs`
**Parser:** `Assets/Axiom/Editor/AgentBridge/Core/JsonCommandParser.cs`

Usage via execute_script (single call, no temp file needed):

```csharp
// Diagnostic tool — returns file path
string result = AgentBridgeGateway.Execute(@"{
    ""tool"": ""reference_scanner"",
    ""mode"": ""scene_references"",
    ""output"": { ""destination"": ""file"" }
}");

// Diagnostic tool — returns report content directly
string result = AgentBridgeGateway.Execute(@"{
    ""tool"": ""hierarchy_lens"",
    ""mode"": ""components"",
    ""scope"": { ""root_path"": ""Player/"", ""max_depth"": 3 },
    ""output"": { ""destination"": ""return"" }
}");

// Action tool
string result = AgentBridgeGateway.Execute(@"{
    ""tool"": ""scene_actions"",
    ""mode"": ""create"",
    ""scope"": { ""object_names"": [""NewEnemy""], ""root_path"": ""Enemies/"" }
}");
```

Mode names accept both descriptive names and letter shortcodes (e.g. `"components"` or `"b"`).

Benefits over direct C# calls:
- No temp script to write, compile, or delete
- Consistent JSON error responses on failure
- Scope filtering applied uniformly across all tools
- Single static method — minimal surface for MCP servers to expose

On error, the gateway returns: `{"error": "description"}`

The gateway now covers ~90% of all operations natively. For operations requiring
complex parameter types (dictionaries, float arrays, struct arrays), the gateway
returns a helpful error message with the exact direct C# call to use as fallback.

For the full command schema, see `project_instructions.md` Section 2.

## Common Discovery Patterns

These patterns replace shell commands and raw file access. Use them instead.

### "Do any [AssetType] assets exist in this project?"
```json
// Via gateway — search by asset type
{"tool": "smart_search", "mode": "assets", "scope": {"asset_extension": ".physicMaterial"}}
```
```csharp
// Or via Unity_RunCommand for exact type queries
AssetDatabase.FindAssets("t:PhysicMaterial")
AssetDatabase.FindAssets("t:AnimatorController")
AssetDatabase.FindAssets("t:ScriptableObject")
```

### "Are there null/missing references on colliders or other components?"
```json
// Reference Scanner — scans all loaded scenes for null references and missing scripts
{"tool": "reference_scanner", "mode": "scene_references"}
```
```json
// Component Inspector Mode F — scoped to specific objects
{"tool": "component_inspector", "mode": "missing_references", "scope": {"root_path": "Player/"}}
```

### "What components are on this GameObject?"
```json
// Hierarchy Lens Mode B — names + component types
{"tool": "hierarchy_lens", "mode": "components", "scope": {"root_path": "Player/"}}
```
```json
// Component Inspector Mode C — full property values for a specific component
{"tool": "component_inspector", "mode": "property_values", "scope": {"root_path": "Player/", "component_filter": "Rigidbody"}}
```

### "What scenes exist in this project?"
```json
// Project Cartographer — find all scene assets
{"tool": "project_cartographer", "mode": "file_manifest", "scope": {"asset_extension": ".unity"}}
```
```json
// Or Smart Search
{"tool": "smart_search", "mode": "assets", "scope": {"asset_extension": ".unity"}}
```

### "What's in a scene that isn't currently loaded?"
```json
// Step 1: Load the scene
{"tool": "scene_actions", "mode": "manage_scene", "scope": {"root_path": "Assets/Scenes/TargetScene.unity", "object_names": ["load"]}}
```
```json
// Step 2: Now inspect it
{"tool": "hierarchy_lens", "mode": "components", "scope": {"scene_name": "TargetScene"}}
```

### "Are there broken shaders or pink materials?"
```json
// Reference Scanner Mode F — material audit
{"tool": "reference_scanner", "mode": "material_audit", "scope": {"asset_path": "Assets/Materials"}}
```

### "Find and remove missing scripts"
```csharp
// Via Unity_RunCommand — one-shot cleanup
GameObjectUtility.RemoveMonoBehavioursWithMissingScript(gameObject);
```

### "What physics settings are configured?"
```json
// Physics Reporter Mode F — full physics settings dump
{"tool": "physics_reporter", "mode": "physics_settings"}
```
```json
// Physics Reporter Mode A — collider census across the scene
{"tool": "physics_reporter", "mode": "collider_census"}
```

### SceneDiff (COMPLETE)
Menu: Axiom > AgentBridge > Scene Diff — (Save Snapshot | Save Snapshot (with Property Values) | Compare Current vs Last Snapshot | List Snapshots | Clear All Snapshots)
Modes: HashOnly (A), ObjectCountDiff (B), StructuralDiff (C), PropertyDiff (D — captures property values in snapshots)
Snapshots stored in: {ProjectRoot}/AgentReports/Snapshots/*.snapshot (manual JSON, not JsonUtility)
Usage:
- `SceneDiff.Execute(SceneDiffOperation.Snapshot, label: "before_player_setup")`
- `SceneDiff.Execute(SceneDiffOperation.Snapshot, mode: SceneDiffMode.PropertyDiff, label: "before_with_props")`
- `SceneDiff.Execute(SceneDiffOperation.CompareCurrent, mode: SceneDiffMode.StructuralDiff, label: "before_player_setup")`
- `SceneDiff.Execute(SceneDiffOperation.CompareCurrent, mode: SceneDiffMode.PropertyDiff, label: "before_with_props")`
- `SceneDiff.Execute(SceneDiffOperation.Compare, mode: SceneDiffMode.HashOnly, labelA: "snap_a", labelB: "snap_b")`
- `SceneDiff.Execute(SceneDiffOperation.List)`
- `SceneDiff.Execute(SceneDiffOperation.Clear)`
Notes:
- JsonUtility is NOT used for snapshot serialization (it drops List<CustomClass> in separate asmdefs).
  A hand-rolled JSON builder + minimal parser is used instead.
- Snapshot labels are used as filenames — avoid special characters.
- Hashing uses SHA-256 over a canonical pipe-delimited string of all entries (sorted by path).
- StructuralDiff detects: added, removed, moved (same name/different path), and component changes.
- PropertyDiff requires both snapshots captured with mode: PropertyDiff.

## Unity MCP Integration (Native)

When the Unity AI Assistant package (com.unity.ai.assistant@2.7+) is installed, Axiom exposes 5 native MCP tools through Unity's built-in MCP bridge. These appear automatically in Edit > Project Settings > AI > Unity MCP.

### Axiom_Gateway (PRIMARY)
The canonical entry point. Forwards JSON commands to AgentBridgeGateway.Execute().
Accepts the same JSON contract documented in project_instructions.md Section 2.
```json
{"tool": "hierarchy_lens", "mode": "components", "scope": {"root_path": "Player/"}, "output": {"destination": "return"}}
```
Returns: {ok: true, data: {tool, mode, destination, result}}

### Axiom_Status
Cheap health probe. Returns editor state, compile status, play mode, report paths.
No parameters.

### Axiom_ReadReport
Reads a generated report from AgentReports/. Use after Axiom_Gateway returns a file path.
Parameters: relativePath OR reportName OR latestPrefix, optional maxChars (default 20000).

### Axiom_Verify
First-class verification tool. Replaces temp-script compilation checks.
Operations: compilation, errors, scene_diff_compare_current
The `compilation` operation automatically calls `AssetDatabase.Refresh(ForceUpdate)` and waits up to 30s for compilation to finish before reporting — ensuring the result reflects the agent's latest file changes, not a stale CLEAN from before.
Example: {"operation": "compilation"}

### Axiom_Rules
Returns compact Axiom operating rules for agents that don't have access to .cursorrules.
No parameters.

### When MCP Tools Are NOT Available
If com.unity.ai.assistant is not installed, these tools don't exist. Use the direct C# approach:
- Write temp scripts to Assets/Axiom/Editor/AgentBridge/Temp/
- Call tools via AgentBridgeGateway.Execute(json) or direct static methods
- Read reports from AgentReports/ via file system
